# Мультитенантная тикет-система поддержки

SaaS-поддержка с изоляцией организаций, ролями, SLA, полнотекстовым поиском и
вложениями. Ядро задачи — **изоляция тенантов на уровне Postgres (RLS)**, а не
в коде приложения.

**Стек:** Next.js 14 (App Router) · TypeScript · Supabase (Postgres + Auth +
Storage) · Tailwind CSS.

---

## Что внутри

| Возможность | Реализация |
|---|---|
| Организации и членство | Таблицы `organizations`, `memberships`; переключение текущей организации через cookie |
| Роли | `owner` / `agent` / `client` — разная видимость одних и тех же таблиц |
| Изоляция | RLS-политики: пользователь видит только данные своих организаций |
| Клиент vs агент | Клиент видит только свои тикеты; агент/владелец — все тикеты организации |
| Внутренние комментарии | `is_internal` виден только сотрудникам — отсекается на уровне `SELECT` в RLS |
| SLA | Дедлайн считается триггером по приоритету; просрочка подсвечивается |
| Поиск | Полнотекстовый `tsvector` (конфигурация `russian`) с ранжированием `ts_rank` |
| Вложения | Приватный Storage-бакет; доступ только участникам по подписанной ссылке |

## Запуск локально

### 1. Создать проект Supabase

Зайдите на [supabase.com](https://supabase.com) → New project. После создания
понадобятся (Project Settings → API):

- `Project URL`
- `anon` (публичный ключ)
- `service_role` (секретный ключ — только для сервера)

### 2. Применить миграции

Откройте **SQL Editor** в дашборде Supabase и выполните файлы по порядку:

```
supabase/migrations/0001_schema.sql
supabase/migrations/0002_functions.sql
supabase/migrations/0003_rls.sql
supabase/migrations/0004_storage.sql
```

Либо через Supabase CLI:

```bash
supabase db push
```

### 3. Отключить подтверждение email (для разработки)

Authentication → Providers → Email → выключить «Confirm email». Иначе после
регистрации придётся подтверждать почту перед входом.

### 4. Переменные окружения

Скопируйте `.env.example` в `.env.local` и заполните:

```bash
NEXT_PUBLIC_SUPABASE_URL=https://<ref>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon-key>
SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
```

### 5. Установка и запуск

```bash
npm install
npm run dev
```

Откройте <http://localhost:3000>. Зарегистрируйте пользователя, создайте
организацию (станете владельцем) — и можно работать.

## Как устроена изоляция (главное)

Вся защита данных живёт **в базе**, а не в приложении. Любой запрос идёт под
JWT пользователя, поэтому даже прямой вызов API/БД подчиняется RLS.

### Helper-функции

`is_org_member`, `is_org_agent`, `is_org_owner`, `role_in_org` объявлены как
`SECURITY DEFINER` — они выполняются от владельца БД и читают `memberships`
**в обход RLS**. Это нужно, чтобы политики на самой таблице `memberships` не
рекурсировали сами на себя (классический паттерн Supabase).

### Политики (примеры логики)

- **tickets / SELECT:** агент и владелец видят все тикеты организации
  (`is_org_agent(org_id)`); клиент — только где он автор
  (`created_by = auth.uid()`).
- **ticket_comments / SELECT:** внутренние комментарии (`is_internal = true`)
  отдаются только сотрудникам. Клиент их физически не получает — фильтровать в
  коде не нужно.
- **organizations / memberships:** видны только те, где пользователь состоит.

### RBAC через триггер

Клиент не может менять статус, приоритет, исполнителя и т.п. — даже если
отправит «сырой» `UPDATE`. Триггер `enforce_ticket_update_rules`
(`SECURITY DEFINER`) откатывает запрещённые для роли изменения. Приложение
лишь прячет элементы управления; финальное слово — за БД.

### SLA

Триггер `compute_sla_due` ставит `sla_due_at` от приоритета:
срочный — 4 ч, высокий — 8 ч, обычный — 24 ч, низкий — 72 ч (от `created_at`).
Пересчитывается при смене приоритета. Просрочка подсвечивается в интерфейсе.

### Поиск

Столбец `search_vector` — `tsvector`, генерируемый из `subject` (вес A) и
`body` (вес B) с конфигурацией `russian` (стемминг для кириллицы). Поиск идёт
через RPC `search_tickets(p_org, p_query)` с `ts_rank` для сортировки по
релевантности. Функция `SECURITY INVOKER` — RLS остаётся в силе, чужие
организации в выдачу не попадают.

### Вложения

Бакет `attachments` приватный (`public = false`) — постоянной публичной ссылки
не существует. Путь: `{org_id}/{ticket_id}/{uuid}-{filename}`. Storage-политики
проверяют членство по первому сегменту пути. Скачивание идёт через
`/api/attachments/[id]`: сервер сначала достаёт строку (под RLS — посторонний
получит 404), затем выдаёт **подписанную ссылку на 60 секунд**.

## Проверка критериев приёмки

**1. Организация A не может получить данные организации B.**
Войдите пользователем из организации A и выполните в браузерной консоли прямой
запрос к таблице тикетов с `org_id` организации B (или просто `select *` без
фильтра). Вернётся пусто: RLS отдаёт только строки организаций пользователя.
Тикет B по прямому URL `/tickets/<id-из-B>` отдаёт 404.

**2. Клиент не видит внутренние комментарии.**
Агент оставляет комментарий с галкой «внутренняя заметка». Клиент-автор тикета
открывает тот же тикет — заметки нет ни в интерфейсе, ни в ответе API
(RLS отсекает на `SELECT`).

**3. Поиск ранжирует по релевантности.**
Создайте несколько тикетов с пересекающимися словами в теме и теле. Поиск по
слову вернёт сначала тикеты, где оно в теме (вес A), затем где только в теле
(вес B) — за счёт `ts_rank`.

**4. Вложение недоступно постороннему по прямой ссылке.**
Прямого URL к файлу нет — бакет приватный. Ссылка `/api/attachments/[id]`
для постороннего вернёт 404 (строку не отдаёт RLS), а выданная подписанная
ссылка живёт 60 секунд.

## Роли

| Действие | owner | agent | client |
|---|:---:|:---:|:---:|
| Видеть все тикеты организации | ✅ | ✅ | только свои |
| Создавать тикеты | ✅ | ✅ | ✅ |
| Менять статус / приоритет / исполнителя | ✅ | ✅ | ❌ |
| Внутренние комментарии (читать/писать) | ✅ | ✅ | ❌ |
| Приглашать участников, менять роли | ✅ | ❌ | ❌ |

## Замечания по безопасности

- `service_role`-ключ используется только на сервере (`lib/supabase/admin.ts`)
  и только для поиска пользователя по email при приглашении. В клиент он не
  попадает.
- Приложение нигде не полагается на проверки доступа «на словах»: убери весь
  фронтенд — изоляция, видимость внутренних комментариев и RBAC продолжат
  работать на уровне Postgres.
